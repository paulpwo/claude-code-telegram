# Security Architecture

This document describes the security system of the `claude-code-telegram` bot for developers who need to understand, configure, or extend it.

---

## 1. Overview — The 5 Layers

Every incoming Telegram message passes through a layered defense stack before Claude executes anything. The layers run in this order:

1. **Authentication** — Verifies the sender is an allowed user before any processing occurs.
2. **Input validation** — Scans the raw message text for dangerous patterns (shell injection, path traversal, suspicious URLs, etc.).
3. **Directory isolation** — Ensures every file path Claude resolves is inside `APPROVED_DIRECTORY`.
4. **Tool monitoring** — Intercepts each tool call Claude attempts at SDK level and blocks boundary violations and unsafe git operations.
5. **Audit logging** — Persists every auth attempt, pattern violation, and tool block to SQLite for forensic review.

Layers 1–2 run as `python-telegram-bot` middleware (handlers registered at groups `-3`, `-2`, and `-1`). Layer 3 is enforced both inside the `SecurityValidator` during path resolution and inside the tool-level callback. Layer 4 is enforced by the `can_use_tool` callback wired into `ClaudeAgentOptions`. Layer 5 is woven throughout all other layers.

## 2. Authentication

### 2.1 Whitelist Auth (`ALLOWED_USERS`)

`WhitelistAuthProvider` in `src/security/auth.py` is the simplest provider. On every authentication attempt it checks whether the Telegram numeric user ID appears in the `allowed_users` set.

```
ALLOWED_USERS=123456789,987654321
```

`ALLOWED_USERS` is a comma-separated list of integer Telegram user IDs. Leave it unset to enable dev mode (`allow_all_dev=True` is derived from an empty list), which grants access to any sender. Dev mode is intended for local development only — never deploy without a populated `ALLOWED_USERS`.

**Admin users.** A separate `ADMIN_USER_IDS` variable designates users who receive both `"basic"` and `"admin"` permissions:

```
ADMIN_USER_IDS=123456789
```

Any user ID in `ADMIN_USER_IDS` that is also in `ALLOWED_USERS` (or in dev mode) will have `permissions: ["basic", "admin"]` in their session. The `admin_required` middleware in `src/bot/middleware/auth.py` checks this list before allowing admin-only commands to proceed.

### 2.2 Token Auth (`ENABLE_TOKEN_AUTH`)

`TokenAuthProvider` supports a second authentication mechanism on top of the whitelist. When enabled, users can be issued a long-lived bearer token and authenticate with it instead of (or in addition to) being in the whitelist.

**Enabling token auth:**

```
ENABLE_TOKEN_AUTH=true
AUTH_TOKEN_SECRET=<at-least-32-random-characters>
```

`AUTH_TOKEN_SECRET` must be at least 32 characters. It is used as the key material for `hashlib.scrypt` — see hashing details below.

**Token issuance.** A new token is generated by calling `TokenAuthProvider.generate_token(user_id)`. The method uses `secrets.token_urlsafe(32)` to produce a 43-character URL-safe random token, hashes it immediately, stores the hash, and returns the plaintext to the caller once. The plaintext is never stored.

**Token storage — `DatabaseTokenStorage`.** By default token data is persisted in the SQLite `user_tokens` table. Tokens survive bot restarts. The storage layer uses an upsert: if the same hash is re-submitted (identical plaintext token), it updates `expires_at` and resets `last_used` rather than inserting a duplicate row.

**Hashing.** Tokens are hashed with `hashlib.scrypt` using the following parameters:

```python
hashlib.scrypt(token.encode(), salt=salt, n=16384, r=8, p=1, dklen=32)
```

The salt is derived from the first 16 bytes of `AUTH_TOKEN_SECRET` (zero-padded if shorter). This is intentionally a cost-tuned KDF, not a fast hash — it makes offline brute-force attacks against a leaked database expensive.

**Expiry.** Tokens expire 30 days after issuance by default. `DatabaseTokenStorage.get_user_token` checks `expires_at` on every lookup and returns `None` for expired tokens.

**Revocation.** Revocation is a soft delete: `DatabaseTokenStorage.revoke_token` sets `is_active = FALSE` on all rows for the given `user_id`. Revoked tokens are excluded from lookups without removing historical data from the table.

### 2.3 Combining providers

Both `WhitelistAuthProvider` and `TokenAuthProvider` can be active simultaneously. `AuthenticationManager` iterates the provider list in order and grants access as soon as the first provider succeeds. There is no requirement for both to pass. In practice this means a user who is in the whitelist is authenticated immediately, regardless of token state.

## 3. Input Validation

Input validation runs in `security_middleware` (`src/bot/middleware/security.py`), which calls `validate_message_content` for text messages and `validate_file_upload` for document uploads.

### 3.1 Classic mode

When `AGENTIC_MODE=false`, any pattern match causes the message to be **rejected** and the user receives an error reply. No further handlers run.

`validate_message_content` checks three categories of patterns inline (independent of `SecurityValidator`):

| Category | Examples |
|----------|---------|
| Command injection | `` `backtick substitution` ``, `$(subshell)`, `; rm`, `&& rm`, `curl ... \| sh`, `wget ... \| sh`, `eval(`, `exec(` |
| Path traversal | `../`, `~/`, `/etc/`, `/var/`, `/usr/`, `/sys/`, `/proc/` |
| Suspicious URLs | `.ru/`, `.tk/`, `.ml/`, `bit.ly/`, `tinyurl.com/`, `javascript:`, `data:text/html` |

After pattern checks the text is passed through `SecurityValidator.sanitize_command_input`, which strips the characters `` `$;|&<>#\x00-\x1f\x7f ``. If more than 50% of the original content is removed, the message is blocked as "Content contains too many dangerous characters".

`SecurityValidator.DANGEROUS_PATTERNS` (used separately for path and filename validation) additionally blocks `..`, `~`, `${`, `$VAR`, `;`, `&&`, `||`, `>`, `<`, single `|`, single `&`, and null bytes.

**To disable pattern blocking in a trusted environment:**

```
DISABLE_SECURITY_PATTERNS=true
```

This flag is passed to `SecurityValidator` at construction time and bypasses the `DANGEROUS_PATTERNS` loop inside `validate_path`. It does not disable the inline checks in `validate_message_content` — those are always active in classic mode.

### 3.2 Agentic mode (`AGENTIC_MODE=true`, default)

In agentic mode, blocking on input validation would break normal usage — legitimate prompts routinely contain backticks, relative paths, pipe characters, and similar constructs.

When `agentic_mode=True`, the security middleware runs the same validation logic but **does not block**. If a pattern match is found, the violation is written to the audit log and a `structlog` entry is emitted, but the message continues to the handler. The enforcement line in agentic mode is the tool-level `can_use_tool` callback (see Section 6), which intercepts what Claude actually tries to execute rather than what the user typed.

## 4. Directory Isolation

`APPROVED_DIRECTORY` is the single root directory that Claude is permitted to read from and write to. It is required at startup — the bot will not start without it.

```
APPROVED_DIRECTORY=/workspace
```

Path validation is performed by `SecurityValidator.validate_path` in `src/security/validators.py`. The method:

1. Resolves the candidate path with `Path.resolve()` (follows all symlinks, collapses `..` sequences).
2. Calls `path.relative_to(approved_directory)` — this raises `ValueError` if the resolved path is not a descendant of `approved_directory`.
3. Returns `(False, None, "Access denied: path outside approved directory")` on failure.

`DANGEROUS_PATTERNS` also blocks explicit traversal tokens (e.g. `..`, `~`) before resolution, providing a defence-in-depth check that fails early before the filesystem is consulted.

**Symlink caveat.** Because validation uses `.resolve()`, a symlink placed inside `APPROVED_DIRECTORY` that points to a target outside it will resolve to that external path and be blocked. Conversely, creating a symlink *into* `APPROVED_DIRECTORY` from an external location and then passing that symlink path to the bot will also resolve to the real path, which may be inside the approved boundary. Do not create symlinks whose resolved targets point outside `APPROVED_DIRECTORY`.

## 5. Webhook Security (if `ENABLE_API_SERVER=true`)

The webhook server is a FastAPI application (`src/api/server.py`) that listens on `API_SERVER_PORT` (default 8080). All webhook payloads arrive at `POST /webhooks/{provider}`.

### GitHub webhooks

Authentication uses HMAC-SHA256 (`src/api/auth.py`, `verify_github_signature`). GitHub signs the raw request body with `GITHUB_WEBHOOK_SECRET` and sends the result as the `X-Hub-Signature-256: sha256=<hex>` header. The server recomputes the expected signature and compares using `hmac.compare_digest` to prevent timing attacks. Requests with a missing, malformed, or incorrect signature are rejected with HTTP 401.

Delivery deduplication uses the `X-GitHub-Delivery` header. Each unique delivery ID is written to the `webhook_events` SQLite table atomically before the event is published to the event bus. Duplicate delivery IDs are silently dropped with HTTP 200 to avoid GitHub retries.

### Generic webhooks

Non-GitHub providers authenticate via a Bearer token in the `Authorization` header (`verify_shared_secret`):

```
Authorization: Bearer <WEBHOOK_API_SECRET>
```

Additionally, a **replay protection window** is enforced. Every request must include an `X-Timestamp` header containing a Unix epoch integer. The server rejects the request if `|now - timestamp| > 300 seconds` (5 minutes). This prevents captured requests from being replayed later.

Deduplication uses the `X-Delivery-ID` header if present. If the header is absent, the server generates a deterministic fallback key:

```python
hashlib.sha256(f"{provider}:{x_timestamp}:{body[:32].hex()}".encode()).hexdigest()
```

This allows deduplication even when the sender does not provide an explicit delivery ID, as long as the payload is identical.

## 6. Tool Monitor

The `can_use_tool` callback (`_make_can_use_tool_callback` in `src/claude/sdk_integration.py`, backed by helpers in `src/claude/monitor.py`) is registered on `ClaudeAgentOptions.can_use_tool`. The SDK invokes it synchronously before executing each tool call, giving the bot a veto point that operates on Claude's actual intentions rather than on user input.

The callback enforces three distinct rules:

**File path boundary.** For tools in `{"Write", "Edit", "Read", "create_file", "edit_file", "read_file"}`, the `file_path` argument is passed to `SecurityValidator.validate_path`. If the resolved path falls outside `APPROVED_DIRECTORY`, the tool call is denied with a `PermissionResultDeny` containing the validation error message.

**Bash directory boundary.** For tools in `{"Bash", "bash", "shell"}`, `check_bash_directory_boundary` tokenises the command with `shlex.split` and examines each command in the pipeline independently. Filesystem-modifying commands (`mkdir`, `touch`, `cp`, `mv`, `rm`, `rmdir`, `ln`, `install`, `tee`, `cd`) have their path arguments resolved and checked against `APPROVED_DIRECTORY`. Read-only commands (`cat`, `ls`, `head`, `tail`, `echo`, etc.) are always allowed through. Unparseable commands are passed through to rely on the OS sandbox.

**Git safety.** For Bash tool calls that start with `git`, `check_git_safety` applies three rules (when `DISABLE_TOOL_VALIDATION=false`):

- `git push --force` / `git push -f` — blocked unless `GIT_ALLOW_FORCE_PUSH=true`
- `git branch -D` — blocked unless `GIT_ALLOW_DELETE_BRANCH=true`
- `git push <remote> <branch>` where `<branch>` is in `GIT_PROTECTED_BRANCHES` — always blocked
- `git reset --hard <ref>` where `<ref>` matches a protected branch name — always blocked

`GIT_PROTECTED_BRANCHES` defaults to `["main", "develop", "master"]`.

**Disabling tool validation:**

```
DISABLE_TOOL_VALIDATION=true
```

When this flag is set, the SDK's allowed/disallowed tool lists are cleared (`None`) so the SDK itself does not restrict tool names, and the git safety checks inside the `can_use_tool` callback are skipped. The directory boundary checks remain active because they run unconditionally inside `_make_can_use_tool_callback` before the `config.disable_tool_validation` guard is reached. Use this flag only in fully trusted environments where MCP tools or other non-standard tools need unrestricted access.

## 7. Audit Logging

`DatabaseAuditStorage` persists security events in the SQLite `audit_log` table. Events are written by:

- `auth_middleware` — every authentication attempt, success or failure.
- `validate_message_content` — command injection attempts, path traversal attempts, suspicious URLs, excessive sanitization.
- `validate_file_upload` — dangerous filenames, oversized files, forbidden MIME types.
- `threat_detection_middleware` — cumulative reconnaissance patterns (more than 5 recon-style commands from the same user).

Each audit log entry captures:

| Field | Description |
|-------|-------------|
| `user_id` | Telegram user ID |
| `event_type` | String identifier (e.g. `"command_injection_attempt"`, `"auth_failure"`) |
| `severity` | `"low"`, `"medium"`, or `"high"` |
| `timestamp` | UTC datetime |
| `details` | Free-text context (pattern matched, filename, etc.) |
| `attempted_action` | What the user was trying to do (e.g. `"message_send"`, `"file_upload"`) |

Because events are written to SQLite, they persist across bot restarts and are available for post-incident review via direct database queries.

| `GIT_ALLOW_DELETE_BRANCH` | `false` | Allow `git branch -D` from Claude. |

## 9. Common Misconfigurations

**Using `DISABLE_SECURITY_PATTERNS=true` in production.**
This flag disables `DANGEROUS_PATTERNS` matching in `validate_path`, which is the primary guard against path traversal in filenames and command arguments. It exists for environments where legitimate paths or commands happen to contain characters like `~` or `$`. In production, the correct fix is to narrow the problematic pattern, not to disable the entire check.

**Not setting `ADMIN_USER_IDS` but registering admin-protected commands.**
The `admin_required` middleware checks `"admin"` in `session.user_info["permissions"]`. If no user IDs are in `ADMIN_USER_IDS`, no session will ever carry the `"admin"` permission, and every call to an admin-protected handler will fail with a permission error — silently, from the developer's perspective, since no error is raised at startup.

**Running the webhook server on a public port without TLS termination.**
The server binds to `API_SERVER_PORT` with no TLS. Deploying it directly on the internet exposes the `Authorization: Bearer` header and request bodies in plaintext. Place the server behind a reverse proxy (nginx, Caddy, Traefik) that handles HTTPS and only forwards traffic to the bot on the loopback interface.

**Symlinking a directory into `APPROVED_DIRECTORY` that resolves outside it.**
Path validation uses `.resolve()` which follows symlinks before checking the boundary. A symlink at `/workspace/external -> /etc` will cause any path under `/workspace/external/` to resolve to `/etc/...`, which is outside the approved directory and will be denied. The risk is the inverse: if you deliberately create a symlink pointing outside `APPROVED_DIRECTORY` expecting it to be blocked, that is correct — it will be. But if you expect a symlink *from* outside to provide access *into* the approved directory, that also works if the resolved path lands inside the boundary.

**Setting a weak `AUTH_TOKEN_SECRET` (fewer than 32 characters).**
The `scrypt` salt is derived from the first 16 bytes of `AUTH_TOKEN_SECRET`. A short or predictable secret reduces the entropy of the salt, weakening the KDF. A compromised `user_tokens` database combined with a guessable secret makes offline token cracking feasible. Generate the secret with a CSPRNG, for example:

```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
```
